技術リサーチ担当、マーケティング分析担当、読者心理分析担当、記事ライター、批判的レビュー担当、最終仕上げ担当。6つのセッションを立ち上げ、調査と分析を並列で走らせた後、その結果を統合して1本の記事に仕上げています。

## Claude Code のマルチエージェントとは

「マルチエージェント」と聞くと難しそうですが、やっていることはチーム開発と同じです。

1人で要件定義・設計・実装・テスト・レビューを全部やるのではなく、**専門の担当者に分担する**。Claude Code のマルチエージェントは、この分担をAIのセッション単位で実現します。

Claude Code には2つのマルチエージェント機能があります。

| 機能 | 仕組み | イメージ |
|------|--------|---------|
| **Task tool（サブエージェント）** | メインセッションから独立したサブエージェントにタスクを委任 | 上司が部下に「これ調べておいて」と頼む |
| **Agent Teams（チーム機能）** | 複数のエージェントを同時に起動し、タスク管理とメッセージングで協調 | プロジェクトチームを編成して並列で動かす |

まずはシンプルな Task tool から見ていきます。

## Task tool: 1対1の委任

Task tool は、メインの Claude Code セッションから**独立したサブエージェント**にタスクを投げる仕組みです。

### 1セッションの限界

以前、Claude Code で[2日でアプリを作った](/articles/claude-code-2days-app)ときは1セッションで全作業を行いました。短い開発なら問題ありませんが、長い記事の執筆やリファクタリングになると、セッション後半でこんな症状が出ます。

- コンテキストが圧縮され、冒頭の指示が反映されなくなる
- 調査フェーズで読んだファイルの情報が薄れる
- 文体やトーンにブレが出る

```
【Before: 1セッションで記事全体を書かせた場合】
- 調査 → 構成 → 執筆 → レビューが直列処理
- 後半になるとコンテキストが圧迫され、冒頭の指示が薄れる
- 調査結果と執筆が混在し、どちらも中途半端に

【After: 複数エージェントで分担した場合】
- 調査・分析・執筆・レビューが独立したコンテキストで進行
- 各エージェントは自分の担当領域に集中
- 調査で100ファイル読んでも、メインセッションには結果だけが返る
```

Task tool を使えば、サブエージェントが**独立したコンテキストウィンドウ**（Claude が一度に参照できる情報量の上限）で作業するので、メインのコンテキストを消費しません。

### 基本の使い方

Task tool には `subagent_type` というパラメータがあり、用途に応じて使い分けます。

| subagent_type | 用途 | 使いどころ |
|---|---|---|
| **Explore** | コードベース探索・深い調査 | ファイル横断検索、アーキテクチャ調査 |
| **Plan** | 計画立案・設計 | 実装計画の策定、リファクタリング戦略 |
| **Bash** | シェルコマンド実行 | ビルド、テスト実行 |
| (未指定) | 汎用タスク | コーディング、ファイル操作全般 |

Claude Code のプロンプトで自然言語で指示するだけで、裏側で Task tool が呼ばれます。並列実行もできるので、「テスト実行」と「リンター実行」を同時に走らせることも可能です。

### Task tool の使い分け判断

```
単純なファイル検索（3クエリ以内で済む）
  → Glob / Grep を直接使う（Task不要）

複数ファイルの横断調査（3クエリ以上）
  → Task (subagent_type: Explore)

実装計画を立ててほしい
  → Task (subagent_type: Plan)

複数の独立した調査を同時にやりたい
  → 複数の Task を並列呼び出し
```

つまり、**コンテキストを分離したい作業や、並列で済ませたい作業**があるときに Task tool の出番です。

Task tool は手軽に使えますが、本格的に複数のエージェントを協調させたい場合は、次に紹介する Agent Teams の出番です。

## Agent Teams: 複数エージェントの協調

Agent Teams は、**複数のエージェントを同時に起動し、タスク管理とメッセージングで連携させる**仕組みです。2026年2月に発表され、現在は experimental フラグ付きで利用できます。

### セットアップ

#### Step 1: 環境変数を設定

`~/.claude/settings.json` に以下を追加します。

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

#### Step 2: エージェントを定義

`~/.claude/agents/` ディレクトリにMarkdownファイルでエージェントを定義します。

```markdown
<!-- ~/.claude/agents/my-developer.md -->
---
name: my-developer
description: 開発担当エージェント
model: sonnet
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
---

# my-developer

あなたは開発担当のエージェントです。
TypeScript / React の開発を行います。
テストを書いてからコードを実装してください。
```

YAML frontmatter で名前・モデル・使えるツールを指定し、本文にシステムプロンプト（行動指針）を書きます。`model` にはエージェントに使わせるモデルを指定します（opus / sonnet / haiku の3段階。コストと性能に応じて選択）。

**ツールの指定がポイントです。** エージェントごとにアクセスできるツールを制限できます。たとえばレビュー担当には `Read` と `Grep` だけ渡して、コードを勝手に変更できないようにする、といった設計ができます。

:::details エージェント定義で指定できる主なツール一覧
| ツールカテゴリ | ツール名 | 典型的な用途 |
|---|---|---|
| 読み取り系 | Read, Glob, Grep | コードレビュー、調査 |
| 書き込み系 | Write, Edit | コーディング、ファイル編集 |
| 実行系 | Bash | ビルド、テスト、コマンド実行 |
| 検索系 | WebSearch | 市場調査、ドキュメント参照 |
| チーム系 | Task, TeamCreate, SendMessage, TaskCreate, TaskUpdate, TaskList | チーム運用・タスク管理 |
| 外部連携 | mcp__* (MCP ツール) | Notion、Slack等との連携 |
:::

#### Step 3: チームを作成

チームリーダー（人間、または司令塔エージェント）が `TeamCreate` でチームを編成します。各メンバーにはエージェント定義と初期プロンプト（何をしてほしいか）を渡します。

### タスク管理: blocks / blockedBy による依存関係

Agent Teams にはタスク管理の仕組みが組み込まれています。

```
TaskCreate  → タスクを作成
TaskUpdate  → ステータス更新、担当者割り当て、依存関係の設定
TaskList    → タスク一覧の確認
```

特に強力なのが **依存関係の管理**です。`blocks` / `blockedBy` を設定することで、「タスクAが完了しないとタスクBを開始できない」というパイプラインを構築できます。

```
Task #1 (リサーチ)  ─┐
Task #2 (分析)     ──┼──→ Task #4 (執筆)  → Task #5 (レビュー)  → Task #6 (公開)
Task #3 (心理分析)  ─┘
```

\#1〜#3 は並列で同時実行され、全て完了した時点で #4 が開始される。**この構成がまさに、この記事の制作で使われたものです。** 実際にこのタスクボードを見ながら、各エージェントが自分のタスクの完了を待ってから次の工程に進む様子を確認できました。

### メッセージング: SendMessage

エージェント間のコミュニケーションには `SendMessage` を使います。

| type | 用途 | いつ使うか |
|------|------|----------|
| `message` | 特定のエージェントへのDM | 「リサーチ完了しました。結果は research-notes.md です」のような報告 |
| `broadcast` | 全メンバーへの一斉送信 | 「全員作業停止、仕様変更があります」のような緊急連絡（コスト高） |
| `shutdown_request` | エージェントへの終了要求 | タスク完了後の後片付け |

**重要な制約として、エージェントのテキスト出力は他のメンバーには見えません。** 他のエージェントに何か伝えたい場合は、必ず `SendMessage` を使う必要があります。人間のチーム開発でSlackを使うのと同じ感覚です。つまり、エージェント同士は「話しかけない限り、お互いの存在を知らない」状態です。

この記事の制作では、リサーチ担当が調査完了後に team-lead に SendMessage で通知し、team-lead がタスクの依存関係を確認してから次の工程（ライター担当）を起動しました。

### 情報共有のベストプラクティス

エージェント間の情報共有には、用途に応じて3つの方法があります。

| 共有方法 | 向いている情報 | この記事での使い方 |
|---------|-------------|----------------|
| **ファイル書き出し** | 調査結果、分析レポート、長文の成果物 | リサーチ担当が `research-notes.md`（620行）に書き出し → ライターが Read |
| **SendMessage** | 完了通知、質問、短い指示 | 各担当が「完了しました。結果は〇〇.md です」と team-lead に報告 |
| **TaskUpdate** | 進捗状況 | ステータスを `completed` に更新 → 依存タスクが自動で開始可能に |

つまり、**長文はファイル、短文はメッセージ、状態管理はタスク**と使い分けるのがポイントです。最初にファイルシステムを使わず SendMessage だけで調査結果を送ろうとしたのですが、受信側のコンテキストを大量に消費してしまうことがわかり、ファイル経由に切り替えました。

## 実践例: この記事の制作プロセス

ここからは、**この記事自体がどうやって作られたか**を公開します。

### チーム構成

```
チーム名: zenn-multiagent-article

team-lead (biz-lead, opus)       ← 司令塔。チーム編成とタスク管理
tech-researcher (dev-builder)    ← 技術リサーチ担当
marketer (biz-analyst)           ← SEO・マーケティング分析担当
psychologist (content-writer)    ← 読者心理分析担当
writer (content-writer)          ← 記事の執筆担当
critic (dev-reviewer)            ← 批判的レビュー担当（Read only）
finalizer (content-writer)       ← 最終仕上げ担当（この工程を担当しているエージェント）
```

team-lead（司令塔）を含めて計7セッションが関わっていますが、同時に起動していたのは最大4セッション（Phase 1 の3リサーチ + team-lead）です。

### ワークフロー

**Phase 1（並列実行）:**

3つのリサーチタスクが同時にスタートしました。

| エージェント | タスク | 出力ファイル | 出力量 |
|------------|--------|------------|--------|
| tech-researcher | Claude Code マルチエージェントの技術仕様調査 | `research-notes.md` | 620行 |
| marketer | SEOキーワード、競合調査、タイトル案 | `marketing-analysis.md` | 343行 |
| psychologist | 読者ペルソナ、感情ジャーニー設計 | `reader-psychology.md` | 365行 |

3つのエージェントが並列で調査・分析を行い、それぞれの結果をMarkdownファイルに書き出しました。

**Phase 2（依存タスク）:**

3つのリサーチが全て完了した時点で、ライター（writer）がそれらを統合して記事を執筆しました。

**Phase 3（レビュー）:**

ライターの原稿を批判的レビュー担当（critic）がチェック。Read と Grep だけのツール構成で、原稿に手を加えることなく、改善提案だけをまとめました。

**Phase 4（最終仕上げ）:**

レビュー結果を反映して最終仕上げを行い、公開準備を完了。今まさにこの工程です。

### 実際に動かしてわかったこと

**うまくいったこと:**
- 3つのリサーチが並列で完了し、ライターは全ての調査結果を参照して記事を書けた。1セッションで「調べながら書く」のとは、情報の網羅性が段違い
- レビュー担当に Write 権限を与えなかったため、指摘事項が明確で修正しやすかった。「ここを直すべき」と「なぜ直すべきか」が分離されるのは想像以上に便利
- 各エージェントの出力が独立したファイルに残るので、人間（自分）が後から確認・修正しやすい

**想定外だったこと:**
- research-notes.md には tech-researcher のモデルが opus と記録されていたが、実際の設定では sonnet のはずだった。エージェント定義のモデルと TeamCreate 時のオーバーライドで混乱が生じることがある
- 全エージェントが opus で動くとAPI費用が跳ね上がる。司令塔のみ opus、他は sonnet にすることでコストを抑えるのが現実的
- エージェント間でファイルの書き込みタイミングが重なると、一時的にファイルの内容が不安定になる場面があった。担当ファイルを明確に分けることが重要

### エージェント定義の設計判断

この記事のチームで意識したポイントを3つ紹介します。

**1. 司令塔だけ opus、他は sonnet**

チームリーダー（team-lead）だけが高コストの opus モデルを使い、実作業を担当するメンバーは sonnet で動いています。opus は高度な判断・統合・チーム運営に向いていますが、調査や執筆は sonnet で十分です。

**2. エージェントごとのツール制限**

レビュー担当（critic / dev-reviewer）には `Read` と `Grep` だけを渡し、`Write` や `Edit` は渡しません。レビュー担当がコードを勝手に書き換えることを防ぐ、最小権限の原則です。

**3. ファイル経由の情報共有**

エージェント間のやり取りは `SendMessage` ではなく、Markdownファイルへの書き出しをメインにしました。理由は2つあります。

- ファイルなら人間もレビューできる
- 長文の調査結果を `SendMessage` で送ると、受信側のコンテキストを圧迫する

## コスト戦略: opus / sonnet / haiku の使い分け

マルチエージェントで最も気になるのがコストです。複数のエージェントが同時にAPIを消費するため、無計画に使うと高額になります。

### モデルの使い分けガイド

| モデル | 入力 / 出力 | 速度 | 向いているタスク |
|-------|-----------|------|----------------|
| **opus** | $15 / $75 per 1M tokens | 遅 | チーム統括、複雑な判断、統合分析 |
| **sonnet** | $3 / $15 per 1M tokens | 中 | コーディング、分析、記事執筆 |
| **haiku** | $0.80 / $4 per 1M tokens | 速 | メトリクス収集、テンプレート適用、SNS投稿ドラフト |

opus は sonnet の5倍のコストです。全員 opus にすると、6エージェント構成で30倍のコスト差になります。

### コスト最適化の実例

私のプロジェクトでは、9種類のエージェントを定義していますが、モデルの割り当てはこうなっています。

```
opus  → biz-lead（司令塔）のみ
sonnet → dev-builder, dev-reviewer, content-writer, biz-analyst, biz-ops, client-manager
haiku  → sns-manager, metrics-collector
```

意思決定と統合判断だけ opus に任せ、実行部隊は sonnet、軽量な定型作業は haiku。これでコストを抑えつつ、品質を維持しています。

:::details この記事の制作コスト（概算）
参考までに、この記事をマルチエージェントで制作した際のAPI費用の目安です（トークン数からの概算）。

| 項目 | 推定費用 |
|------|---------|
| team-lead (opus) -- チーム管理・タスク設計 | $2-4 |
| リサーチ3本 (sonnet x 3) -- 調査・分析 | $1-3 |
| 記事執筆 (sonnet) | $0.5-1 |
| レビュー (sonnet) | $0.3-0.5 |
| 最終仕上げ (sonnet) | $0.3-0.5 |
| **合計** | **$4-9** |

1セッション（opus）で全部やった場合は、長いコンテキストを維持するため $5-10 程度かかると推定されます。マルチエージェントの方がやや高くなる可能性がありますが、品質（専門性の分離）と速度（並列処理）を考えると、用途次第で十分元が取れます。

全員 opus にした場合は $20-40 以上になるため、モデルの使い分けが重要です。
:::

### 不要なエージェントは終了させる

タスクが完了したエージェントは `shutdown_request` で終了させましょう。起動したまま放置すると、無駄なAPI消費が発生します。

## 注意点と制限事項

マルチエージェントは万能ではありません。実際に使ってみてわかった注意点をまとめます。

### 同時ファイル編集の競合

複数のエージェントが同じファイルを同時に編集すると、競合が発生します。対策は2つ。

- ファイルの担当を明確に分ける（エージェントAは `src/` 、エージェントBは `tests/`）
- レビュー担当は Read only にして書き込み権限を外す

### コンテキストは共有されない

各エージェントは独立したコンテキストウィンドウを持っています。「リサーチ担当が読んだファイルの内容を、ライター担当が知っている」ということはありません。情報はファイルか SendMessage で明示的に渡す必要があります。

### experimental フラグ

2026年3月時点では `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` という環境変数が必要です。仕様が変更される可能性があるため、プロダクションのCI/CDに組み込む際は注意してください。

### すべてをマルチエージェントにする必要はない

単純なバグ修正や1ファイルの変更に、わざわざチームを編成する必要はありません。**タスクが複数の専門領域にまたがる場合**や、**並列で処理できる独立したサブタスクがある場合**に、マルチエージェントが力を発揮します。

判断基準をシンプルにまとめると:

```
1ファイルの修正、単純な質問
  → 通常の Claude Code セッションで十分

複数ファイルの横断調査（3クエリ以上）
  → Task tool でサブエージェントに委任

複数の専門領域にまたがるプロジェクト
  → Agent Teams でチームを編成
```

## まとめ

Claude Code のマルチエージェントは、**1つのセッションに全部詰め込む開発スタイルからの脱却**を可能にします。

この記事自体が、6体のAI + 1人の人間というチーム構成で制作されました。

### マルチエージェントがやったこと

- 技術仕様の調査（research-notes.md: 620行）
- SEO・競合分析（marketing-analysis.md: 343行）
- 読者心理設計（reader-psychology.md: 365行）
- 上記3つを統合した記事の執筆
- 批判的レビューと改善提案（13項目の指摘）
- レビュー反映と最終仕上げ

### 人間（自分）がやったこと

- チーム構成の設計（誰に何をやらせるか）
- エージェント定義の作成
- 最終的な記事の品質判断と修正
- 公開の判断

マルチエージェントで変わったのは「手を動かす」部分です。何を書くか、誰に読ませるか、品質は十分かの判断は、依然として人間の仕事です。

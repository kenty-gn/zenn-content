## この記事のポイント

- 2026年3月、JetBrainsがコーディングエージェント「Junie CLI」のベータ版を公開した
- Claude Codeを毎日使っている個人開発者の視点で、Junie CLIとの違いを整理する
- 「どっちが優れてるか」ではなく「どういう人にどっちが合うか」を考える

## 背景

Claude Codeを使い倒して個人開発をしている毎日。ターミナルを開いてClaude Codeに話しかけるのが日常になっています。

そんな中、2026年3月にJetBrainsが「Junie CLI」をベータ公開しました。

> Junie CLI, the LLM-agnostic coding agent, is now in Beta

公式ブログを読んだ第一印象は「これ、Claude Codeのガチ競合じゃん」。

同時に「JetBrains Air」というエージェント特化の新IDE（廃止されたFleetの後継）も発表され、JetBrainsがコーディングエージェント市場に本気で参入してきたことが伝わってきます。

気になったので、Claude Codeユーザーの目線で比較調査してみました。

## やったこと

公式ブログ、比較記事、技術レビューを一通り読んで、個人開発者として重要なポイントを整理しました。

### 基本スペックの比較

**Junie CLI**

- **モデル**: GPT-5.2, Claude Sonnet/Opus, Gemini 3 Pro, Grok等を**自由に切替**（LLM非依存）
- **コンテキスト**: 200kトークン
- **動作環境**: ターミナル / IDE / CI・CD / GitHub / GitLab
- **価格**: $100/年（JetBrains AI Pro）or BYOK（自分のAPIキー持ち込み・追加課金なし）

**Claude Code**

- **モデル**: Claude専用（Opus 4.6 / Sonnet 4.6）
- **コンテキスト**: **400kトークン**（Junieの2倍）
- **動作環境**: ターミナル / IDE / Slack / ブラウザ
- **価格**: 従量課金 or Max契約（$100/月〜）

### Junie CLIの強み

**1. LLM非依存 — モデルを自由に切り替えられる**

これがJunie CLIの最大の売り。タスクに応じてモデルを選べます。

- 単純な修正 → 安いモデル（Gemini Flash等）でコスト節約
- 複雑な設計 → Claude Opusで品質重視
- 特定領域 → そのタスクに強いモデルを選択

ただし**注意点**。モデルを切り替えるとエージェントのコンテキスト（学習内容）がリセットされます。「さっきの続きをGPTでやって」ができない。これは現時点での大きな制約です。

https://memu.pro/blog/junie-cli-model-agnostic-coding-memory

**2. BYOK（Bring Your Own Key）**

自分のAPIキーを持ち込めば、JetBrainsへの追加課金なし。すでにClaude APIキーを持っている人は実質無料。これは強い。

**3. 次タスク予測**

プロジェクトの文脈を理解して「次にやるべきこと」を先回りで提案してくれます。Claude Codeにはない機能。

**4. MCP自動検出**

必要なMCPサーバーを自動で検出・推薦し、ワンクリックでインストール。Claude Codeだと手動で設定ファイルを書く必要があるので、これは楽。

**5. ワンクリック移行**

Claude CodeやCodexからの設定移行がワンクリック。乗り換えのハードルが極めて低い。

### Claude Codeの強み

**1. 400kコンテキスト — 大規模プロジェクトの安心感**

Junie CLIの2倍。10ファイル以上にまたがるリファクタリングや、モノレポでの作業ではこの差が効いてきます。

個人開発でも、認証・DB・UI・課金をまたいだ横断的な変更をするときに「コンテキストに入りきらない」ストレスがないのは大きい。

**2. CLAUDE.md — プロジェクトのルールブック**

`.claude/CLAUDE.md`にプロジェクト固有のルールや指示を書いておけば、毎回の指示なしにプロジェクトの文脈を理解してくれます。

自分の場合、コーディング規約や仮想カンパニーの運営ルールまでCLAUDE.mdに書いていて、これがないとワークフローが成り立たない。

**3. チェックポイント / ロールバック**

生成結果が壊れた場合にファイルシステムを即座に巻き戻せる。「やっぱり元に戻して」が一瞬。何度も救われてます。

**4. Anthropic直営の最適化**

モデルとエージェントが密結合で最適化されている。Claude Opus 4.6の能力をフルに引き出す設計になっているのは、専用エージェントならではの強み。

## 結果と数字

スペック比較をまとめると:

- コンテキスト: **Claude Code 400k** vs Junie 200k
- モデル柔軟性: Claude Code 1社のみ vs **Junie 複数社対応**
- 年間コスト: Claude Code $1,200〜（Max） vs **Junie $100/年 or BYOK**
- Claude Code年間収益: **$25億**（2026年2月時点）
- エコシステム規模: Claude Code > Junie（現時点）

## 学んだこと

### 「Claude Code一択」の時代は終わりつつある

JetBrainsが本気でコーディングエージェント市場に参入してきた。LLM非依存というアプローチは、ベンダーロックインを嫌うエンジニアにとって魅力的。

### でも、大事なのはワークフローとの相性

ツールの性能差より、自分の開発スタイルにフィットするかどうかが重要。

**Claude Codeが合う人**

- Claude Opus/Sonnetの品質に満足している
- 大規模なコンテキストが必要（モノレポ、多ファイルプロジェクト）
- CLAUDE.mdでプロジェクトルールを管理したい
- チェックポイントの安心感が欲しい

**Junie CLIが合う人**

- 複数のLLMを使い分けたい（コスト最適化）
- JetBrains IDEをメインで使っている
- CI/CDにAIエージェントを組み込みたい
- すでにAPIキーを持っていて追加課金したくない

### 個人的な結論

現時点では**Claude Codeを継続**。理由は3つ。

1. **400kコンテキストの安心感** — 個人開発でも10ファイル以上触るリファクタはよくある
2. **CLAUDE.mdが深く組み込まれている** — 仮想カンパニー運用まで含めてCLAUDE.mdに依存
3. **チェックポイントに何度も救われている** — 特にUI変更で「やっぱり前のほうが良かった」が多い

### LLM切替の課題

Junie CLIの最大の売りであるLLM切替だけど、コンテキストがリセットされる問題は致命的。これが解決されたら評価が大きく変わりそう。

# 第9章: リリース後に稼ぐ --- ASO・収益改善・グロース戦略

App Store に公開できた。おめでとうございます。しかし、ここからが本番です。

本書のタイトルは「稼げるアプリを作る実践ガイド」です。第8章までで「作る」は完了しました。この章では「稼ぐ」ために必要な、公開後の初動施策・ASO・収益指標の読み方・改善サイクルを解説します。

## 9.1 リリース直後の72時間でやること

App Store に公開された直後は、アルゴリズム上の「新着ブースト」が働きます。この期間を最大限に活かすための施策を整理します。

### 初日にやること

1. **SNSで告知する**: X（Twitter）、Zenn、個人ブログなど自分の発信チャネルすべてで告知。App Storeのリンクを添える
2. **知人にレビューを依頼する**: 初期レビューの数と星の平均がランキングに直結する。5件以上を目標に
3. **スクリーンショットとプレビュー動画を最終確認**: 公開後でも App Store Connect から更新可能

### 初週にやること

1. **App Store Connect のアナリティクスを確認**: インプレッション数・プロダクトページ閲覧数・ダウンロード数の3つを毎日チェック
2. **クラッシュレポートの監視**: Xcode Organizer または App Store Connect の「クラッシュ」タブで致命的な問題がないか確認
3. **ユーザーからの問い合わせ対応**: 初期ユーザーの声は金脈。改善のヒントが詰まっている

:::message
**ふくログの実例**: 初週のダウンロード数は、SNS告知をしたかどうかで大きく変わりました。Xでの告知投稿に App Store のリンクとスクリーンショットを添えるだけで、オーガニック検索だけの場合と比べてダウンロード数に明確な差が出ます。
:::

## 9.2 App Store 最適化（ASO）の基本

ASO（App Store Optimization）は、App Store 内の検索でアプリを見つけてもらうための最適化です。SEO の App Store 版と考えてください。

### タイトルとサブタイトル

App Store のタイトル（30文字以内）とサブタイトル（30文字以内）は、検索ランキングに最も影響する要素です。

```
// ふくログの例
タイトル:    ふくログ - 副業収入管理
サブタイトル: 確定申告に備える収支記録アプリ
```

**ポイント:**
- タイトルにはアプリ名 + 主要キーワードを含める
- サブタイトルにはユーザーが検索しそうな言葉を含める（「確定申告」「収支記録」など）
- 競合アプリのタイトルを調査し、差別化できるキーワードを選ぶ

### キーワードフィールド

App Store Connect の「キーワード」フィールド（100文字以内）には、タイトルやサブタイトルに含めなかったキーワードを設定します。

```
// キーワード設定のコツ
- カンマ区切りでスペースなし（文字数節約）
- タイトル・サブタイトルと重複するキーワードは不要（自動で含まれる）
- 一般的すぎるキーワード（「アプリ」等）は避ける
- ユーザーが実際に検索する具体的なフレーズを選ぶ
```

### スクリーンショットの最適化

App Store のプロダクトページで最初に目に入るのはスクリーンショットです。

**効果的なスクリーンショットの構成:**

1. **1枚目**: アプリの最大の価値を伝えるメイン画面（ダッシュボード等）
2. **2枚目**: 主要機能のハイライト（収入記録画面など）
3. **3枚目**: 差別化ポイント（20万円進捗バーなど、独自機能）
4. **4〜6枚目**: その他の機能や設定画面

各スクリーンショットには短いキャッチコピーを重ねると、スクロールだけで機能が伝わります。Figma や Canva でテンプレートを作っておくと、アップデート時も効率的に更新できます。

### 説明文の書き方

App Store の説明文（4,000文字以内）は検索ランキングには直接影響しませんが、ダウンロードの意思決定に影響します。

```
// 効果的な説明文の構造
1行目: アプリの価値を一文で伝える（最初の3行だけが折りたたみ前に表示される）
2-3行目: 具体的なメリットを箇条書き

[主な機能]
・機能A — 説明
・機能B — 説明
・機能C — 説明

[こんな方におすすめ]
・ターゲットユーザー像

[料金]
・無料で使える範囲
・Pro プランの内容と価格（自動更新の説明を含める）
```

## 9.3 RevenueCat ダッシュボードで追うべき指標

第6章で RevenueCat の設定は完了しています。ここでは、公開後にダッシュボードで何を見るべきかを解説します。

### 4つの重要指標

RevenueCat の Overview 画面で、以下の4指標を定期的にチェックします。

| 指標 | 意味 | 目安 |
|------|------|------|
| **MRR**（Monthly Recurring Revenue） | 月次経常収益。現在の有料ユーザー × 月額 | 個人開発の最初の目標は MRR 1万円 |
| **Active Trials** | 無料トライアル中のユーザー数 | トライアルを提供している場合に追跡 |
| **Conversion Rate** | トライアル → 有料への転換率 | 業界平均は 10〜30% |
| **Churn Rate** | 月次解約率 | 10% 以下を目指す |

### RevenueCat Charts の読み方

RevenueCat の **Charts** タブでは、時系列でのトレンドを確認できます。

```
// 特に注目すべきチャート
1. Revenue: 日次・週次・月次の収益推移
2. Active Subscriptions: 有料ユーザー数の推移
3. Churn: 解約数と解約率の推移
4. Trial Conversion: トライアルからの転換推移
```

**実運用のコツ**: 毎週月曜日に RevenueCat ダッシュボードを確認する習慣をつけましょう。数字が小さいうちは日次で見ても変動が大きすぎて判断できません。週次トレンドで見るのがおすすめです。

### Supabase 側で見るユーザー行動

RevenueCat は課金の指標に特化しています。「ユーザーがアプリをどう使っているか」は Supabase 側で確認します。

```sql
-- アクティブユーザー数（過去7日間にデータを追加したユーザー）
SELECT COUNT(DISTINCT user_id)
FROM income_records
WHERE created_at > NOW() - INTERVAL '7 days';

-- ユーザーあたりの平均レコード数（エンゲージメントの指標）
SELECT
  user_id,
  COUNT(*) as record_count
FROM income_records
GROUP BY user_id
ORDER BY record_count DESC;

-- 登録後に1件もレコードを追加していないユーザー（オンボーディング改善の対象）
SELECT p.id, p.created_at
FROM profiles p
LEFT JOIN income_records i ON p.id = i.user_id
WHERE i.id IS NULL
AND p.created_at < NOW() - INTERVAL '3 days';
```

これらのクエリを Supabase の SQL Editor で定期的に実行することで、「登録はしたけど使っていないユーザー」や「ヘビーユーザー」の傾向が見えてきます。

## 9.4 ペイウォール最適化 --- コンバージョン率を上げる

第6章で実装したペイウォール（Pro 機能の制限画面）は、収益に最も直結する UI です。

### ペイウォール表示のタイミング

コンバージョン率を左右する最大の要因は、**いつペイウォールを表示するか**です。

```
// タイミング別の効果
❌ 初回起動直後 → ユーザーがアプリの価値を理解する前。離脱率が高い
❌ 設定画面の奥 → そもそも見てもらえない
✅ Pro 機能を使おうとしたとき → 「これが欲しい」と思った瞬間に提示
✅ 無料版で価値を実感した直後 → レコードを5件登録した後など
```

ふくログの場合、月次レポートや年間サマリーなどの集計機能を Pro 限定にしています。ユーザーが収入・経費を数件入力して「もっと詳しく見たい」と感じたタイミングでペイウォールが表示される設計です。

### 価格表示のベストプラクティス

RevenueCat の Offerings で設定した月額・年額を表示する際のポイントです。

```tsx
// 年額プランの訴求例
<View>
  <Text>月額プラン</Text>
  <Text>¥490/月</Text>
</View>

<View>
  {/* 年額を月額換算で表示すると割安感が伝わる */}
  <Text>年額プラン</Text>
  <Text>¥4,900/年（月額換算 ¥408）</Text>
  <Badge>17%おトク</Badge>
</View>
```

**年額プランを推す理由**: 年額ユーザーは月額ユーザーに比べて LTV（顧客生涯価値）が高く、チャーン率も低い傾向があります。ペイウォール上で年額プランを視覚的に目立たせる（色を変える、「おすすめ」バッジを付ける）のが定石です。

### A/Bテスト: RevenueCat Experiments

RevenueCat の **Experiments** 機能を使うと、異なるペイウォールのバリエーションをテストできます。

```
// テストできる要素の例
- 価格（月額 480円 vs 490円 vs 580円）
- 年額の割引率（17% vs 25% vs 33%）
- ペイウォールのコピー（機能訴求 vs 価格訴求）
- 無料トライアルの有無と期間（3日 vs 7日 vs なし）
```

RevenueCat ダッシュボードの **Experiments** タブから Offering を複数作成し、ユーザーをランダムに振り分けます。統計的に有意な差が出るまで（通常100〜300件のイベントが必要）テストを継続します。

:::message alert
A/B テストは十分なユーザー数がないと有意な結論が出ません。DAU（日次アクティブユーザー）が100人を超えるまでは、感覚ベースで改善を繰り返す方が効率的です。
:::

## 9.5 解約（チャーン）を防ぐ

新規ユーザーの獲得よりも、既存ユーザーの維持の方がコストが低く、収益への貢献度が高い。チャーン対策は個人開発の収益を安定させる鍵です。

### 解約タイミングを知る

第7章で実装した RevenueCat Webhook では、`CANCELLATION` と `EXPIRATION` のイベントを処理しています。`CANCELLATION` は「自動更新をオフにした」タイミング、`EXPIRATION` は「実際にサブスクが切れた」タイミングです。

この2つのイベントの間に**グレースピリオド**があります。ユーザーが `CANCELLATION` した後、サブスク期間が終わるまでの間にリテンション施策を打てる可能性があります。

### アプリ内でのリテンション施策

```tsx
// 解約予定ユーザーへのメッセージ例
// subscriptionStore の willRenew フラグで判定
const WillExpireBanner = () => {
  const { willRenew, expirationDate } = useSubscriptionStore();

  if (willRenew !== false) return null;

  return (
    <View style={styles.banner}>
      <Text>
        Pro プランは {formatDate(expirationDate)} に終了します。
        継続すると月次レポートや年間サマリーを引き続きご利用いただけます。
      </Text>
      <Button
        title="プランを確認する"
        onPress={() => Linking.openURL(
          'https://apps.apple.com/account/subscriptions'
        )}
      />
    </View>
  );
};
```

### レビュー依頼で満足度を可視化する

`expo-store-review` を使って、適切なタイミングでレビューを依頼します。

```bash
npx expo install expo-store-review
```

```tsx
import * as StoreReview from 'expo-store-review';

// レビュー依頼のタイミング例
const requestReviewIfAppropriate = async (recordCount: number) => {
  // 条件: レコードが10件以上 & まだ依頼していない
  if (recordCount >= 10) {
    const isAvailable = await StoreReview.isAvailableAsync();
    if (isAvailable) {
      await StoreReview.requestReview();
    }
  }
};
```

:::message
Apple のガイドラインでは、レビューダイアログの表示は年に3回までに制限されています。ユーザーがアプリの価値を実感したタイミング（一定数のレコード登録後など）で1回だけ表示するのがベストプラクティスです。
:::

## 9.6 副業20万円の壁を超える収益計算

ふくログのターゲットユーザーは副業で収入を得ている人です。そして著者自身も、このアプリで「副業としての収益」を目指しています。具体的な数字で考えてみましょう。

### 損益分岐点の計算

```
// ふくログの収益構造
月額サブスク: 490円（Apple の手数料 15% を引くと 416円）
年額サブスク: 4,900円（手数料後 4,165円 → 月額換算 347円）

// 固定費
Apple Developer Program: 12,800円/年（月額換算 1,067円）
Supabase: 無料枠（MAU 50,000まで）
RevenueCat: 無料枠（MRR $2,500まで）
ドメイン（プライバシーポリシー用）: 約1,500円/年

// 月間固定費合計: 約1,200円
```

### 収益シミュレーション

```
// 月額ユーザーのみの場合（手数料後 416円/人・月）
損益分岐:   3人 × 416円 = 1,248円/月 ≥ 固定費 1,200円
月1万円:   24人 × 416円 = 9,984円/月
月5万円:  120人 × 416円 = 49,920円/月
年20万円:  40人 × 416円 × 12ヶ月 = 199,680円 ≈ 20万円
```

**年間20万円を超えるには、月額ユーザー40人を維持すればよい**。これは決して不可能な数字ではありません。

### コストが増えるタイミング

```
// Supabase の無料枠を超えるケース
Database: 500MB まで無料 → 1ユーザーあたり約1MB想定で500人まで
Auth: MAU 50,000 まで無料 → 個人開発では十分
Edge Functions: 500,000 呼び出し/月まで無料
Storage: 1GB まで無料

// RevenueCat の有料プランが必要になるケース
MRR $2,500（約37万円）を超えたら → 月額 $119〜
→ 個人開発でここに到達したら嬉しい悲鳴
```

個人開発のスモールスタートでは、Supabase と RevenueCat の無料枠で十分にカバーできます。コストを心配するよりも、まずユーザー獲得に集中しましょう。

## 9.7 撤退か継続か --- 月次レビューのフレームワーク

個人開発で最も難しいのは「続けるか、やめるか」の判断です。感情ではなく、数字で判断するためのフレームワークを紹介します。

### 月次レビューで確認する5項目

毎月1回、以下の5項目を確認します。

| # | 項目 | 確認方法 | 判断基準 |
|---|------|---------|---------|
| 1 | 新規ダウンロード数 | App Store Connect | 先月比で減少傾向が3ヶ月続いていないか |
| 2 | 有料ユーザー数 | RevenueCat Dashboard | 純増しているか（新規 > 解約） |
| 3 | チャーン率 | RevenueCat Charts | 月次 15% を超えていないか |
| 4 | ユーザーからのフィードバック | App Store レビュー・X | 改善すべき具体的な要望があるか |
| 5 | 開発にかけた時間 | 自分の記録 | 時給換算で最低賃金を下回り続けていないか |

### 判断の目安

```
📈 継続のサイン:
  - MRR が月次で 5% 以上成長している
  - ユーザーから具体的な機能要望が来ている
  - ASO 改善やペイウォール最適化の余地がまだある

⚠️ ピボットのサイン:
  - 3ヶ月連続でダウンロード数が減少
  - チャーン率が改善施策を打っても下がらない
  - ユーザーの声が「使いにくい」ではなく「そもそも必要ない」

🛑 撤退のサイン:
  - 6ヶ月以上 MRR が横ばいで、改善のアイデアが尽きた
  - Apple Developer Program の年会費を回収できる見込みがない
  - 自分自身がアプリに情熱を持てなくなった
```

### 撤退しても残るもの

仮にアプリの収益化がうまくいかなかったとしても、以下の資産は残ります。

- **技術スタック**: Expo + Supabase + RevenueCat を使った実装経験
- **App Store 公開実績**: ポートフォリオとして活用可能
- **コードベース**: 次のアプリの基盤として再利用可能
- **知見**: 何がうまくいかなかったかの学び

個人開発は「すべてのアプリで成功する」必要はありません。試行回数を増やすことが、最終的に収益化につながるアプリに出会う確率を上げます。
